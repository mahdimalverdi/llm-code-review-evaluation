## One-Sentence Summary

> P31 introduces CodeJudgeBench, a benchmark for LLM-as-a-judge on code generation, code repair, and unit-test generation, showing that reasoning/thinking judges perform best but still suffer from randomness, position bias, and limited generalization.

## Main Contribution

The paper creates a challenging coding-focused judge benchmark with 5,352 curated good/bad response pairs and evaluates 26 judge models. It highlights that previous judge benchmarks are too easy for frontier models and that coding judges need stronger robustness testing.

## Dataset / Study Context

| Field | Value |
|---|---|
| Benchmark | CodeJudgeBench |
| Samples | 5,352 curated pairs |
| Source problems | LiveCodeBench-v6, 1,055 coding competition problems from May 2023–April 2025 |
| Tasks | Code generation, code repair, unit test generation |
| Pair format | Instruction, good response, bad response |
| Response generators | Claude, Gemini, Qwen and other strong models |
| Judge models | 26 LLM-as-a-judge models, open and closed, thinking and non-thinking |

## Method

| Component | Details |
|---|---|
| Code generation | Good/bad code based on unit-test pass/fail. |
| Code repair | Erroneous code + error message + two repair candidates; good/bad by tests. |
| Test generation | Candidate input-output tests; correctness by ground-truth output comparison. |
| Difficulty split | Easy/medium/hard based on judge success rates. |
| Pairwise evaluation | Correct response alternated between A and B to test position sensitivity. |
| Prompt experiments | Pairwise vs pointwise; raw response vs full code vs no comments; BoN. |

## Key Findings

| Finding | Summary |
|---|---|
| F1 | Thinking/reasoning judge models outperform non-thinking models. |
| F2 | Small thinking models such as Qwen3-8B can outperform much larger judge-tuned models. |
| F3 | Fine-tuning thinking models specifically as judges does not always improve performance. |
| F4 | Closed-source Gemini-2.5-Pro and Claude-4-Sonnet achieve the strongest performance. |
| F5 | Unit-test generation is hardest to judge, followed by code generation; code repair is easiest. |
| F6 | Judge performance can vary by up to about 14% when the correct response is swapped between position A and B. |
| F7 | Judges generalize poorly across responses generated by different coding models. |
| F8 | Pointwise judging underperforms pairwise judging because many pointwise scores tie. |
| F9 | Using the full raw response performs better than stripping to code-only; comments/reasoning help judges. |
| F10 | Pairwise BoN performance is affected by position bias and robustness. |

## Problematic Judge / Evaluation Types

- Position-biased judge.
- Source-model-sensitive judge.
- Non-thinking judge near random baseline.
- Judge-tuned model that does not transfer to coding judgment.
- Pointwise judge that overproduces ties.
- Code-only judge input that removes useful comments/reasoning.
- Judge that performs well on easy benchmarks but fails on challenging subtle pairs.
- BoN judge whose ranking is undermined by pairwise bias.

## Context-Quality Evidence

P31 shows that judging code is not only about seeing executable code. Comments, explanations, and reasoning in raw responses can help the judge decide correctness. This is a useful warning against aggressive preprocessing in evaluation pipelines.

## Trade-off Extraction

| Strategy | Benefit | Risk / Cost |
|---|---|---|
| Thinking judge | Better coding judgment and self-verification. | Higher inference cost and still not fully robust. |
| Pairwise judging | Better than pointwise for subtle code correctness. | Strong position/order bias. |
| Pointwise judging | Simpler scalar scoring. | Many ties and lower discrimination. |
| Raw-response judging | Preserves useful reasoning/comments. | May introduce style/verbosity bias. |
| Code-only judging | Reduces noise and style effects. | Removes useful explanatory cues and lowers performance. |
| Recent benchmark problems | Reduces contamination. | More difficult and potentially less comparable to older work. |

## Relevance to Our Paper

P31 is strong evidence that any LLM-as-a-judge evaluation for code review must test robustness, not just report agreement or accuracy. We should include order-swap checks, prompt-format checks, source-model sensitivity, and preprocessing sensitivity.

## Limitations from Our Perspective

- Focuses on code generation/repair/test generation, not review comments.
- Uses execution-free judge decisions against unit-test-derived labels, which may not capture readability, maintainability, or review usefulness.
- Pairwise setup may not reflect multi-issue PR review.
- Strong emphasis on correctness, less on socio-technical usefulness.
