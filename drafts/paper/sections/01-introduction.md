# Introduction

Modern code review is a central practice in software engineering. Beyond detecting defects, it supports maintainability, knowledge sharing, team awareness, and shared ownership of code. Empirical studies of modern code review show that developers value review not only because it can identify faults, but also because it improves understanding, encourages discussion, and helps teams align on design and implementation decisions [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews]. These broader functions make code review a socio-technical activity rather than a purely mechanical quality-control step. They also imply that an automated review assistant should not be evaluated only by defect-oriented metrics or direct acceptance signals, because review value can include learning, coordination, and maintainability benefits [@p28_heander2025_support_not_automation; @p39_bosu2015_useful_reviews; @p40_ram2018_reviewability].

Recent advances in large language models have renewed interest in automating or supporting code review. LLM-based systems can generate review comments, summarize changes, suggest fixes, classify review feedback, and assist with secure or static-analysis-guided review [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Recent work has also moved beyond isolated code hunks toward pull-request-level, context-aware, and production-informed settings [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These developments make LLM-based review assistance increasingly plausible, but they also make evaluation more difficult: generated comments must be assessed not only as text, but as review interventions inside a development workflow.

Early evaluations often relied on lexical or reference-based metrics, but generated review comments are difficult to evaluate with a single reference. A comment can be useful even when it does not match a human reference in wording, and a comment can be lexically plausible while being unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. This difficulty is amplified by the fact that human review data is not a perfect ground truth: review comments may be noisy, incomplete, shaped by reviewer experience, or difficult to interpret outside their original workflow context [@p08_liu2025_too_noisy; @p23_lin2026_reviewer_experience; @p42_wasserbaech2024_chatgpt_github]. Recent work has therefore introduced richer evaluation criteria, including usefulness, grounding, actionability, hallucination, context alignment, and workflow impact [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p10_sun2025_bitsai_cr; @p27_chowdhury2026_industry_claims]. However, these criteria are often studied in isolation. A benchmark may emphasize issue coverage, a hallucination detector may focus on unsupported claims, a production system may track acceptance or resolution, and an LLM-as-a-Judge study may focus on evaluator agreement. Each view is useful, but none is sufficient on its own.

Consider a generated review comment that warns about a possible null dereference in a changed method. A post-generation verifier may suppress the comment because the provided diff does not contain enough evidence to establish the defect. This decision reduces unsupported warnings, but it may also remove a useful signal if the missing evidence exists in another file, a project-level invariant, or a runtime contract. Conversely, showing the comment as-is may burden reviewers with an uncertain claim. This example illustrates why evaluation should measure not only whether a generated comment is wrong, but also whether a mitigation decision preserves useful feedback at an acceptable cost.

The central challenge is that evaluation decisions involve trade-offs. A stricter post-generation verifier may reduce unsupported comments, but it may also suppress comments that are potentially useful but weakly grounded [@p02_tantithamthavorn2026_hallujudge; @p35_mcaleese2024_llm_critics]. Adding more context may improve grounding, but it can also increase cost, latency, noise, and attention dilution [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p16_icoz2026_context_aware]. A relevance or actionability filter may reduce reviewer burden, but it may remove early design concerns or comments that are useful after rewriting [@p07_olewicki2024_revmate; @p10_sun2025_bitsai_cr; @p39_bosu2015_useful_reviews]. LLM-based evaluators can scale assessment, but they can also introduce prompt sensitivity, order effects, judge-choice dependence, and other measurement risks [@p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. Therefore, evaluating LLM-generated review comments requires more than asking whether a comment is correct or whether a filter removes bad comments. It also requires asking what useful feedback is preserved, what review coverage is lost, what cost is introduced, and how reliable the evaluator is.

This paper argues that the main limitation of current evaluation practice is fragmentation. Prior work has studied quality metrics, hallucination, context enrichment, data curation, production signals, human usefulness, and evaluator validity [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p03_tantithamthavorn2026_rovodev; @p08_liu2025_too_noisy; @p11_zhang2025_laura; @p29_wang2025_human_evaluators]. However, these dimensions are rarely connected to the decision that an LLM-based review assistant must ultimately support: whether a generated comment should be shown, suppressed, rewritten, or escalated. As a result, existing evaluations can report that a method reduces unsupported comments, improves grounding, or increases acceptance, but they often do not reveal what useful feedback is lost, what review coverage is reduced, what human or computational cost is introduced, or how reliable the evaluator is. This makes it difficult to compare systems across studies and to reason about deployment trade-offs.

To address this gap, we develop a framework-oriented synthesis of LLM-based code review evaluation. We first synthesize evidence from recent and foundational work on automated code review, human code review, context-aware review, production deployments, and LLM-as-a-Judge evaluation [@p14_li2022_codereviewer; @p37_sadowski2018_google_mcr; @p39_bosu2015_useful_reviews; @p03_tantithamthavorn2026_rovodev; @p33_he2025_llmjudge_se]. From this evidence, we derive an operational taxonomy of problematic LLM-generated review comments. The taxonomy distinguishes, for example, unsupported or hallucinated comments, irrelevant comments, incorrect technical claims, wrong-location comments, non-actionable comments, low-value nitpicks, invalid fix suggestions, context-dependent cases, security false alarms, and unsupported performance claims [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p46_zhou2025_vulnerability_repair; @p50_peng2025_coffe]. The goal is not only to name these failures, but to make them annotatable through definitions, decision rules, and examples.

We then propose a multi-dimensional evaluation framework that connects comment quality to context quality, usefulness, actionability, mitigation decisions, cost, workflow impact, and evaluator validity. The framework treats context quality as something to be evaluated, not merely as an input setting, drawing on evidence that reviewability, retrieved context, specifications, documentation, and project-level information can affect whether a comment is judgeable and useful [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr; @p40_ram2018_reviewability; @p49_lee2025_metamon]. It also treats LLM-based judging as a measurement instrument that requires calibration rather than as ground truth [@p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. To make the taxonomy and framework more concrete, we define an annotation protocol and evaluation schema for a small annotated evidence layer. This layer is intended to support the framework by showing how problematic comment types, usefulness, groundedness, actionability, and show/suppress/rewrite/escalate decisions can be labeled and analyzed.

This paper makes four contributions. First, it synthesizes evidence from LLM-based code review, human code review, context-aware review, and LLM-as-a-Judge studies to identify fragmentation in current evaluation practice. Second, it proposes an operational taxonomy of problematic LLM-generated code review comments, designed for annotation rather than only conceptual classification. Third, it introduces a trade-off-aware evaluation framework that connects error reduction, useful-feedback preservation, review coverage, cost, context quality, and evaluator validity to concrete mitigation decisions. Fourth, it defines an annotation protocol and evaluation schema for a small evidence layer that can be used to examine problematic-comment distributions, annotator agreement, and trade-offs among showing, suppressing, rewriting, and escalating generated comments.
