# Introduction

A generated review comment is not just something to score after the fact. Once it is shown to a developer, it becomes part of the review workflow: it can help, distract, mislead, or simply create more work. This matters because modern code review is not only a defect-detection practice. It also supports maintainability, knowledge sharing, team awareness, and shared ownership of code [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. A review assistant therefore has to be evaluated as a workflow participant, not only as a text generator [@p28_heander2025_support_not_automation; @p40_ram2018_reviewability].

Large language models have made automated and semi-automated code review more practical. LLM-based systems can generate review comments, summarize changes, suggest fixes, classify feedback, support secure review, and combine generated feedback with static-analysis or project-level context [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Recent work has also moved toward more realistic settings, including pull-request-level benchmarks, context-aware review assistance, and production-informed systems [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These advances make LLM-based code review more promising, but they also make evaluation harder: the question is no longer only whether a model can produce plausible feedback, but whether that feedback should reach a reviewer in the first place.

The practical difficulty is the presence of problematic generated review comments. We use this term for comments that are technically incorrect, unsupported by the available context, irrelevant to the change, linked to the wrong location or cause, non-actionable, invalid as fix suggestions, or too low-value to justify reviewer attention. Such comments can reduce developer trust, increase review burden, trigger unnecessary rework, or force humans to verify claims that should not have been shown. Removing bad comments is also not enough. A mitigation strategy can suppress useful weak signals, reduce automatic review coverage, increase latency, add model calls, or move difficult decisions back to human reviewers.

Existing evaluations cover important parts of this problem, but they often leave these trade-offs implicit. Lexical and reference-based metrics are limited because a generated review comment can be useful without matching a single human reference, while a fluent comment can still be unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. Hallucination and grounding studies focus on unsupported claims [@p02_tantithamthavorn2026_hallujudge], context-aware approaches study richer evidence [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr], production systems report workflow signals [@p03_tantithamthavorn2026_rovodev; @p10_sun2025_bitsai_cr], and LLM-as-a-Judge work studies evaluator scalability and reliability [@p29_wang2025_human_evaluators; @p33_he2025_llmjudge_se]. For our purpose, the missing connection is between measurement and action: fewer studies compare mitigation strategies on the same review instances while measuring both the errors removed and the useful feedback lost.

Consider a generated comment warning about a possible null dereference in a changed method. A post-generation verifier may suppress it because the local diff does not contain enough evidence. That can reduce unsupported warnings, but it can also remove a useful signal if the relevant evidence sits in another file, a project-level invariant, or a runtime contract. Showing the comment as-is has the opposite risk: it asks reviewers to spend time checking an uncertain claim. A context-quality gate may route the case to a human instead of generating a comment, but then automatic review coverage drops. This small example captures the central trade-off of the paper: mitigation strategies should be evaluated not only by how many problematic comments they remove, but also by how many useful comments they preserve, how much coverage remains, and what cost is introduced.

This paper studies mitigation as an empirical trade-off problem. We define a paired controlled evaluation in which representative mitigation strategies are applied to the same set of code-review instances. The planned strategies include a baseline LLM reviewer, robust prompting, a context-quality gate, post-generation verification, and a hybrid gate-plus-verifier design. The goal is not to declare one universally best method. Instead, we ask which failure types each strategy reduces and how each strategy affects useful-feedback preservation, review coverage, human escalation, and execution cost.

To support this comparison, we define an operational taxonomy of problematic LLM-generated review comments and an annotation protocol that separates technical correctness, grounding, relevance, usefulness, actionability, context quality, dataset validity, and mitigation decision. The initial taxonomy is derived from prior work and is intended to be refined during pilot annotation. It supports labels such as unsupported or hallucinated comment, irrelevant comment, wrong location or wrong cause, invalid fix suggestion, non-actionable comment, low-value comment, and context-dependent case [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer]. The annotation protocol maps comments to concrete workflow decisions: show, suppress, rewrite, or escalate.

These choices lead to research questions about failure-type reduction, useful-feedback preservation, review coverage, human escalation, cost, and context-quality effects. The study is intentionally bounded. It uses a targeted literature review to define the design space, but it is not a full systematic literature review. It is designed as a controlled empirical comparison, but it is not a leaderboard benchmark or model-ranking paper. Its contribution is to make mitigation trade-offs measurable and visible for LLM-generated code review comments.

At the current stage, the paper makes four design-stage contributions. First, it defines an initial operational taxonomy of problematic LLM-generated code review comments for human annotation and strategy evaluation. Second, it introduces a trade-off-aware evaluation schema that measures error reduction together with useful-feedback preservation, review coverage, human escalation, computational cost, context quality, and evaluator validity. Third, it defines a controlled empirical comparison protocol for evaluating representative mitigation strategies on shared review instances. Fourth, it frames mitigation as a workflow decision problem, showing how show, suppress, rewrite, and escalate decisions can expose costs that single correctness, acceptance, or hallucination metrics may hide.
