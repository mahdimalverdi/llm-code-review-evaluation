# Introduction

Modern code review is a central practice in software engineering. It helps teams find defects, but its value goes beyond defect detection. It also supports maintainability, knowledge sharing, team awareness, and shared ownership of code. Empirical studies and surveys show that developers value review because it improves understanding, encourages discussion, and helps teams align on design and implementation decisions [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. These broader functions make code review a socio-technical activity, not only a mechanical quality-control step. They also mean that an automated review assistant should not be evaluated only by defect-oriented metrics or direct acceptance signals. Review value may also come from learning, coordination, and maintainability benefits [@p28_heander2025_support_not_automation; @p39_bosu2015_useful_reviews; @p40_ram2018_reviewability].

Recent advances in large language models have renewed interest in supporting code review with automation. Code review automation is not a new problem. Earlier work showed that automation can reduce human effort, but also that aggregate performance scores may hide important limitations [@p52_tufano2021_automating_code_review_activities; @p53_tufano2024_code_review_automation_strengths_weaknesses]. LLM-based systems extend this line of work. They can generate review comments, summarize changes, suggest fixes, classify review feedback, and assist with secure or static-analysis-guided review [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Recent studies have also moved beyond isolated code hunks toward pull-request-level, context-aware, and production-informed settings [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These developments make LLM-based review assistance more realistic. At the same time, they make evaluation harder. A generated review comment should not be judged only as text; it should be judged as an intervention in a development workflow.

Early evaluations often relied on lexical or reference-based metrics. These metrics are limited because generated review comments are difficult to evaluate with a single reference. A comment can be useful even when it does not match a human reference in wording. Conversely, a comment can look plausible while being unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. Human review data is also not a perfect ground truth. Review comments can be noisy, incomplete, shaped by reviewer experience, or difficult to interpret outside their original workflow context [@p08_liu2025_too_noisy; @p23_lin2026_reviewer_experience; @p42_wasserbaech2024_chatgpt_github]. For this reason, recent work has introduced richer evaluation criteria, such as usefulness, grounding, actionability, hallucination, context alignment, and workflow impact [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p10_sun2025_bitsai_cr; @p27_chowdhury2026_industry_claims]. These criteria are valuable, but they are often studied separately. A benchmark may emphasize issue coverage. A hallucination detector may focus on unsupported claims. A production system may track acceptance or resolution. An LLM-as-a-Judge study may focus on evaluator agreement. Each view is useful, but none is sufficient on its own.

Consider a generated review comment that warns about a possible null dereference in a changed method. A post-generation verifier may suppress the comment because the provided diff does not contain enough evidence to establish the defect. This decision reduces unsupported warnings. However, it may also remove a useful signal if the missing evidence exists in another file, a project-level invariant, or a runtime contract. Showing the comment as-is has the opposite risk: it may burden reviewers with an uncertain claim. This example shows why evaluation should measure more than whether a generated comment is wrong. It should also measure whether a mitigation decision preserves useful feedback at an acceptable cost.

The central challenge is that evaluation decisions involve trade-offs. A stricter post-generation verifier may reduce unsupported comments, but it may also suppress comments that are potentially useful but weakly grounded [@p02_tantithamthavorn2026_hallujudge; @p35_mcaleese2024_llm_critics]. Adding more context may improve grounding, but it can also increase cost, latency, noise, and attention dilution [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p16_icoz2026_context_aware]. A relevance or actionability filter may reduce reviewer burden, but it may remove early design concerns or comments that could be useful after rewriting [@p07_olewicki2024_revmate; @p10_sun2025_bitsai_cr; @p39_bosu2015_useful_reviews]. LLM-based evaluators can scale assessment, but they can introduce prompt sensitivity, order effects, judge-choice dependence, and other measurement risks [@m04_zheng2023_llm_judge; @p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. Therefore, evaluating LLM-generated review comments requires more than asking whether a comment is correct or whether a filter removes bad comments. It also requires asking what useful feedback is preserved, what review coverage is lost, what cost is introduced, and how reliable the evaluator is.

This paper argues that the main limitation of current evaluation practice is fragmentation. Prior work has studied quality metrics, hallucination, context enrichment, data curation, production signals, human usefulness, and evaluator validity [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p03_tantithamthavorn2026_rovodev; @p08_liu2025_too_noisy; @p11_zhang2025_laura; @p29_wang2025_human_evaluators]. However, these dimensions are rarely connected to the decision that an LLM-based review assistant must support: whether a generated comment should be shown, suppressed, rewritten, or escalated. As a result, existing evaluations can report that a method reduces unsupported comments, improves grounding, or increases acceptance. They often do not show what useful feedback is lost, what review coverage is reduced, what human or computational cost is introduced, or how reliable the evaluator is. The novelty of this paper is not another isolated evaluation metric or mitigation method. It is a decision-oriented framework that connects evaluation signals to mitigation decisions and their trade-offs.

To address this gap, we develop a framework-oriented synthesis of LLM-based code review evaluation. Methodologically, the paper is a focused evidence synthesis with a framework-oriented goal. It is complemented by an annotation protocol and a small illustrative evidence layer, rather than by a large-scale benchmark. We first synthesize evidence from recent and foundational work on automated code review, human code review, context-aware review, production deployments, and LLM-as-a-Judge evaluation [@p14_li2022_codereviewer; @p37_sadowski2018_google_mcr; @p39_bosu2015_useful_reviews; @p03_tantithamthavorn2026_rovodev; @p33_he2025_llmjudge_se]. From this evidence, we derive an operational taxonomy of problematic LLM-generated review comments. The taxonomy includes unsupported or hallucinated comments, irrelevant comments, incorrect technical claims, wrong-location comments, non-actionable comments, low-value nitpicks, invalid fix suggestions, and context-dependent cases [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer]. The goal is not only to name these failures. It is to make them annotatable through definitions, decision rules, and examples. The taxonomy is therefore intended to be operational rather than merely descriptive, following the need for explicit dimensions, decision rules, and ending conditions in taxonomy development [@m01_nickerson2013_taxonomy].

We then propose a multi-dimensional evaluation framework. The framework connects comment quality to context quality, usefulness, actionability, mitigation decisions, cost, workflow impact, and evaluator validity. It treats context quality as something to be evaluated, not merely as an input setting. This choice is motivated by evidence that reviewability, retrieved context, specifications, documentation, and project-level information can affect whether a comment is judgeable and useful [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr; @p40_ram2018_reviewability; @p49_lee2025_metamon]. The framework also treats LLM-based judging as a measurement instrument that needs calibration, not as ground truth [@m04_zheng2023_llm_judge; @p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. To make the taxonomy and framework concrete, we define an annotation protocol and evaluation schema for a small annotated evidence layer. This layer shows how problematic comment types, usefulness, groundedness, actionability, and show/suppress/rewrite/escalate decisions can be labeled and analyzed.

This paper makes four contributions. First, it synthesizes evidence from LLM-based code review, human code review, context-aware review, and LLM-as-a-Judge studies to identify fragmentation in current evaluation practice. Second, it proposes an operational taxonomy of problematic LLM-generated code review comments, designed for annotation rather than only conceptual classification. Third, it introduces a trade-off-aware evaluation framework that connects error reduction, useful-feedback preservation, review coverage, cost, context quality, and evaluator validity to concrete mitigation decisions. Fourth, it defines an annotation protocol and evaluation schema for a small evidence layer that can be used to examine problematic-comment distributions, annotator agreement, and trade-offs among showing, suppressing, rewriting, and escalating generated comments.
