# Introduction

A generated review comment can be cheap to produce and expensive to handle. Once it appears in a pull request, a developer has to decide whether to trust it, check it, ignore it, or act on it. In that sense, the comment is no longer just a text output to be scored after the fact. It becomes part of the review workflow, where it can help, distract, mislead, or simply create more work.

This workflow view matters because modern code review is not only a defect-detection practice. It also supports maintainability, knowledge sharing, team awareness, and shared ownership of code [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. A review assistant should therefore be evaluated as a participant in that workflow, not only as a generator of plausible review text [@p28_heander2025_support_not_automation; @p40_ram2018_reviewability].

Large language models have made automated and semi-automated code review more practical. Recent systems can generate review comments, suggest fixes, classify feedback, support secure review, and use static-analysis or project-level context [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. The setting has also become more realistic: recent work studies pull-request-level benchmarks, context-aware review assistance, and production-informed review systems [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These advances make LLM-based code review more useful to study, but they also make evaluation harder. The important question is not only whether a model can produce plausible feedback. It is whether that feedback should reach a reviewer in the first place.

The difficulty becomes clear when generated comments are problematic. In this paper, a problematic generated review comment is one that is technically incorrect, unsupported by the available context, irrelevant to the change, linked to the wrong location or cause, non-actionable, invalid as a fix suggestion, or too low-value to justify reviewer attention. Such comments can reduce developer trust, increase review burden, trigger unnecessary rework, or force humans to verify claims that should not have been shown.

Simply removing bad comments does not solve the problem either. A strict filter can also suppress weak but useful signals. A verifier can improve grounding but add latency and model calls. A context gate can avoid unreliable automatic review, but it may push more cases back to human reviewers. The evaluation problem is therefore not just about finding bad comments. It is about understanding what is lost, preserved, delayed, or escalated when a mitigation strategy intervenes.

Existing evaluations cover important parts of this problem, but the connection between measurement and action remains under-specified. Lexical and reference-based metrics are limited because a generated review comment can be useful without matching a single human reference, while a fluent comment can still be unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. Other strands of work address different pieces of the same challenge. Hallucination studies help identify unsupported claims [@p02_tantithamthavorn2026_hallujudge], context-aware approaches examine what additional evidence can add [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr], production systems report workflow signals [@p03_tantithamthavorn2026_rovodev; @p10_sun2025_bitsai_cr], and LLM-as-a-Judge work studies scalable evaluation and its reliability limits [@p29_wang2025_human_evaluators; @p33_he2025_llmjudge_se]. These pieces are useful, but they do not yet show how mitigation strategies behave on the same concrete review instances when we measure both the errors they remove and the useful feedback they lose.

Consider a generated comment warning about a possible null dereference in a changed method. A post-generation verifier may suppress it because the local diff does not contain enough evidence. That decision may reduce unsupported warnings. It may also remove a useful signal if the relevant evidence sits in another file, a project-level invariant, or a runtime contract. Showing the comment as-is creates the opposite risk: reviewers must spend time checking an uncertain claim. A context-quality gate may route the case to a human instead of generating a comment, which may be safer but reduces automatic review coverage. The same comment can therefore lead to several reasonable decisions, each with a different cost.

We study this problem as an empirical trade-off. The proposed evaluation applies representative mitigation strategies to the same set of code-review instances. The planned strategies include a baseline LLM reviewer, robust prompting, a context-quality gate, post-generation verification, and a hybrid gate-plus-verifier design. Rather than looking for one universally best method, we ask which failure types each strategy reduces and how each strategy affects useful-feedback preservation, review coverage, human escalation, and execution cost.

To make that comparison possible, we define an operational taxonomy of problematic LLM-generated review comments and an annotation protocol for judging them. The taxonomy separates technical correctness, grounding, relevance, usefulness, actionability, context quality, dataset validity, and mitigation decision. It is derived from prior work and is intended to be refined during pilot annotation. The initial labels include unsupported or hallucinated comment, irrelevant comment, wrong location or wrong cause, invalid fix suggestion, non-actionable comment, low-value comment, and context-dependent case [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer]. The annotation protocol then maps comments to concrete workflow decisions: show, suppress, rewrite, or escalate.

The study is intentionally bounded. It uses a targeted literature review to define the design space, but it is not a full systematic literature review. It is designed as a controlled empirical comparison, not as a leaderboard benchmark or a model-ranking paper. The contribution is to make mitigation trade-offs visible: which errors are removed, which useful signals are preserved or lost, how much coverage remains, and when the safest decision is to rewrite or escalate rather than simply show or suppress.

In its current design-stage form, the paper contributes three connected artifacts. First, it defines an initial taxonomy for problematic LLM-generated review comments. Second, it introduces a trade-off-aware evaluation schema for error reduction, useful-feedback preservation, review coverage, human escalation, computational cost, context quality, and evaluator validity. Third, it specifies a controlled comparison protocol for representative mitigation strategies on shared review instances. Together, these artifacts frame mitigation as a workflow decision problem rather than a binary filtering problem.
