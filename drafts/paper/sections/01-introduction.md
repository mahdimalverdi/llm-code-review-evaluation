# Introduction

Modern code review is a central software engineering practice. It helps teams find defects, but its value goes beyond defect detection: review also supports maintainability, knowledge sharing, team awareness, and shared ownership of code [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. This socio-technical role matters for automation. A generated review comment is not only a text artifact to score; it is an intervention in a workflow that can help, distract, mislead, or shift effort back to human reviewers [@p28_heander2025_support_not_automation; @p40_ram2018_reviewability].

Large language models have renewed interest in automating and supporting code review. LLM-based systems can generate review comments, summarize changes, suggest fixes, classify feedback, support secure review, and combine generated feedback with static-analysis or project-level context [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Recent work has also moved toward more realistic settings, including pull-request-level benchmarks, context-aware review assistance, and production-informed systems [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These advances make LLM-based code review more promising, but they also make evaluation harder.

A key practical problem is the presence of problematic generated review comments. We use this term for comments that are technically incorrect, unsupported by the available context, irrelevant to the change, linked to the wrong location or cause, non-actionable, invalid as fix suggestions, or too low-value to justify reviewer attention. Such comments can reduce developer trust, increase review burden, trigger unnecessary rework, or force humans to verify claims that should not have been shown. At the same time, removing problematic comments is not sufficient by itself. A mitigation strategy may also suppress useful weak signals, reduce automatic review coverage, increase latency, add model calls, or move difficult decisions to human reviewers.

Existing evaluations cover important parts of this problem but often leave these trade-offs implicit. Lexical and reference-based metrics are limited because generated review comments can be useful without matching a single human reference, while fluent comments can still be unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. Hallucination and grounding studies focus on unsupported claims [@p02_tantithamthavorn2026_hallujudge], context-aware approaches study richer evidence [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr], production systems report workflow signals [@p03_tantithamthavorn2026_rovodev; @p10_sun2025_bitsai_cr], and LLM-as-a-Judge work studies evaluator scalability and reliability [@p29_wang2025_human_evaluators; @p33_he2025_llmjudge_se]. However, fewer studies compare mitigation strategies on the same review instances while measuring both error reduction and what is lost in the process.

Consider a generated comment warning about a possible null dereference in a changed method. A post-generation verifier may suppress the comment because the diff does not contain enough evidence. This may reduce unsupported warnings, but it may also remove a useful signal if the relevant evidence exists in another file, a project-level invariant, or a runtime contract. Showing the comment as-is has the opposite risk: it may burden reviewers with an uncertain claim. A context-quality gate may route the case to a human rather than generate a comment, but this reduces automatic review coverage. This example illustrates the central trade-off studied in this paper: mitigation strategies should be evaluated not only by how many problematic comments they remove, but also by how many useful comments they preserve, how much coverage remains, and what cost is introduced.

This paper therefore studies mitigation as an empirical trade-off problem. We design a paired controlled evaluation in which representative mitigation strategies are applied to the same set of code-review instances. The strategies include a baseline LLM reviewer, robust prompting, a context-quality gate, post-generation verification, and a hybrid gate-plus-verifier design. The goal is not to declare one universally best method. Instead, we ask which failure types each strategy reduces and how each strategy affects useful-feedback preservation, review coverage, human escalation, and execution cost.

To support this comparison, we develop an operational taxonomy of problematic LLM-generated review comments and an annotation protocol that separates technical correctness, grounding, relevance, usefulness, actionability, context quality, dataset validity, and mitigation decision. The taxonomy is derived from prior work and refined for annotation. It supports labels such as unsupported or hallucinated comment, irrelevant comment, wrong location or wrong cause, invalid fix suggestion, non-actionable comment, low-value comment, and context-dependent case [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer]. The annotation protocol maps comments to concrete decisions: show, suppress, rewrite, or escalate.

These goals lead to research questions about failure-type reduction, useful-feedback preservation, review coverage, human escalation, cost, and context-quality effects. The study is intentionally bounded. It uses a targeted literature review to define the design space, but it is not a full systematic literature review. It uses a controlled empirical comparison, but it is not a leaderboard benchmark or model-ranking paper. Its contribution is to make mitigation trade-offs measurable and visible for LLM-generated code review comments.

This paper makes four contributions. First, it defines an operational taxonomy of problematic LLM-generated code review comments for use in human annotation and strategy evaluation. Second, it introduces a trade-off-aware evaluation schema that jointly measures error reduction, useful-feedback preservation, review coverage, human escalation, computational cost, context quality, and evaluator validity. Third, it defines a controlled empirical comparison protocol for evaluating representative mitigation strategies on shared review instances. Fourth, it analyzes mitigation as a workflow decision problem, showing how show, suppress, rewrite, and escalate decisions expose costs that single correctness, acceptance, or hallucination metrics can hide.
