# Introduction

Modern code review is a central practice in software engineering. It helps teams find defects, but its value goes beyond defect detection: review also supports maintainability, knowledge sharing, team awareness, and shared ownership of code [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. This socio-technical role matters for automation. If review value includes learning, coordination, and maintainability, then an automated review assistant should not be evaluated only by defect-oriented metrics or direct acceptance signals [@p28_heander2025_support_not_automation; @p40_ram2018_reviewability].

Recent advances in large language models have renewed interest in supporting code review with automation. Code review automation is not new: earlier work already showed both the promise of reducing human effort and the risk that aggregate scores can hide important limitations [@p52_tufano2021_automating_code_review_activities; @p53_tufano2024_code_review_automation_strengths_weaknesses]. LLM-based systems extend this line of work by generating review comments, summarizing changes, suggesting fixes, classifying review feedback, and assisting with secure or static-analysis-guided review [@p14_li2022_codereviewer; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Recent studies also examine richer settings, including pull-request-level, context-aware, and production-informed review assistance [@p04_kumar2026_swe_prbench; @p06_hu2025_contextcrbench; @p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These developments make LLM-based review assistance more realistic, but they also make evaluation harder: a generated review comment is not merely a text output, but an intervention in a development workflow.

Current evaluation practice remains fragmented. Lexical and reference-based metrics are limited because generated review comments can be useful without matching a single human reference, while fluent comments can still be unsupported, irrelevant, or non-actionable [@p01_lu2025_deepcrceval; @p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback]. Human review data is also an imperfect ground truth because review comments can be noisy, incomplete, shaped by reviewer experience, or difficult to interpret outside their original workflow context [@p08_liu2025_too_noisy; @p23_lin2026_reviewer_experience; @p42_wasserbaech2024_chatgpt_github]. Recent work has introduced richer criteria such as usefulness, grounding, actionability, hallucination, context alignment, and workflow impact [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p10_sun2025_bitsai_cr; @p27_chowdhury2026_industry_claims]. However, these criteria are often studied separately: benchmarks may emphasize coverage, hallucination detectors may focus on unsupported claims, production systems may track acceptance, and LLM-as-a-Judge studies may focus on evaluator agreement.

Consider a generated review comment that warns about a possible null dereference in a changed method. A post-generation verifier may suppress the comment because the provided diff does not contain enough evidence to establish the defect. This reduces unsupported warnings, but it may also remove a useful signal if the missing evidence exists in another file, a project-level invariant, or a runtime contract. Showing the comment as-is has the opposite risk: it may burden reviewers with an uncertain claim. This example illustrates why evaluation should measure more than whether a generated comment is wrong. It should also ask whether a mitigation decision preserves useful feedback at an acceptable cost.

The central challenge is therefore not only comment correctness, but trade-off-aware decision making. A mitigation strategy may reduce unsupported comments while losing useful weak signals; added context may improve grounding while increasing cost or noise; filtering may reduce reviewer burden while reducing review coverage; and LLM-based evaluation may scale assessment while introducing measurement bias [@p02_tantithamthavorn2026_hallujudge; @p06_hu2025_contextcrbench; @p07_olewicki2024_revmate; @p11_zhang2025_laura; @p29_wang2025_human_evaluators; @p33_he2025_llmjudge_se]. Existing evaluations can report that a method improves grounding, reduces hallucination, or increases acceptance, but they often do not show what useful feedback is lost, what review coverage is reduced, what cost is introduced, or how reliable the evaluator is. The novelty of this paper is not another isolated metric or mitigation method. It is a decision-oriented framework that connects evaluation signals to mitigation decisions and their trade-offs.

To address this gap, we develop a framework-oriented synthesis of LLM-based code review evaluation. The paper is a focused evidence synthesis with a framework-oriented goal, complemented by an annotation protocol and a small illustrative evidence layer rather than by a large-scale benchmark. We synthesize evidence from automated code review, human code review, context-aware review, production deployments, and LLM-as-a-Judge evaluation [@p14_li2022_codereviewer; @p37_sadowski2018_google_mcr; @p39_bosu2015_useful_reviews; @p03_tantithamthavorn2026_rovodev; @p33_he2025_llmjudge_se]. From this evidence, we derive an operational taxonomy of problematic LLM-generated review comments, including unsupported or hallucinated comments, irrelevant comments, incorrect technical claims, wrong-location comments, non-actionable comments, low-value comments, invalid fix suggestions, and context-dependent cases [@p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification; @p21_peng2025_icodereviewer]. The taxonomy is intended to be operational rather than merely descriptive: it supports annotation through definitions, decision rules, and examples, following the need for explicit dimensions and ending conditions in taxonomy development [@m01_nickerson2013_taxonomy].

We then propose a multi-dimensional evaluation framework. The framework connects comment quality to context quality, usefulness, actionability, mitigation decisions, cost, workflow impact, dataset validity, and evaluator validity. It treats context quality as something to be evaluated, not merely as an input setting, because reviewability, retrieved context, specifications, documentation, and project-level information can affect whether a comment is judgeable and useful [@p06_hu2025_contextcrbench; @p11_zhang2025_laura; @p12_wang2025_sgcr; @p40_ram2018_reviewability; @p49_lee2025_metamon]. It also treats LLM-based judging as a measurement instrument that needs calibration, not as ground truth [@m04_zheng2023_llm_judge; @p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. The illustrative evidence layer shows how problematic-comment types, usefulness, groundedness, actionability, and show/suppress/rewrite/escalate decisions can be labeled and analyzed.

This paper makes four contributions. First, it synthesizes evidence from LLM-based code review, human code review, context-aware review, and LLM-as-a-Judge studies to identify fragmentation in current evaluation practice. Second, it proposes an operational taxonomy of problematic LLM-generated code review comments, designed for annotation rather than only conceptual classification. Third, it introduces a trade-off-aware evaluation framework that connects error reduction, useful-feedback preservation, review coverage, cost, context quality, dataset validity, and evaluator validity to concrete mitigation decisions. Fourth, it defines an annotation protocol and evaluation schema for a small evidence layer that can be used to examine problematic-comment distributions, annotator agreement, and trade-offs among showing, suppressing, rewriting, and escalating generated comments.
