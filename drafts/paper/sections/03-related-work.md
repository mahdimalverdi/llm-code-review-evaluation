# Related Work

This section positions the proposed taxonomy and trade-off-aware evaluation framework within prior work on code review automation, LLM-based review assistance, review-comment evaluation, context grounding, data quality, human-centered usefulness, mitigation strategies, and evaluator validity. The aim is not to summarize the papers one by one. Instead, we use prior work to build the evaluation gap that motivates this paper: a review assistant must decide whether each generated comment should be shown, suppressed, rewritten, escalated, or used only as internal evidence.

This section is not intended to be an exhaustive systematic review of all LLM-for-software-engineering work. It focuses on the strands of literature that directly affect evaluation decisions for generated code review comments: what is evaluated, what context is available, how usefulness is judged, what mitigation decision follows, what useful feedback may be lost, and how reliable the evaluator is.

The literature has made substantial progress on individual parts of this problem. Recent work has introduced richer code-review rubrics, hallucination and grounding checks, PR-level benchmarks, context-aware review generation, industrial filters, data curation methods, human-in-the-loop studies, static-analysis hybrids, and LLM-as-a-Judge protocols. However, these strands are still only partially connected. They often evaluate whether a generated comment is plausible or whether a model performs well on a benchmark, but they less often evaluate the full decision pipeline: the quality of the input context, the type of comment failure, the usefulness of the feedback, the risk of losing useful comments under filtering, the cost of mitigation, and the reliability of the evaluator.

## Human Code Review and Review Automation Foundations

Modern code review is not only a defect-detection activity. Empirical studies show that code review also supports maintainability, knowledge transfer, team awareness, shared ownership, alternative-solution discovery, accountability, and developer learning [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. This matters for LLM-based review because a generated comment can be technically correct while still being a poor use of reviewer attention, and a comment can be valuable even when it does not lead directly to an accepted patch. Useful review feedback is therefore a developer- and workflow-centered concept, not merely a factuality label.

Pre-LLM work on automating code review established review-comment generation and review-driven code transformation as concrete tasks. CodeReviewer framed code review as a sequence-to-sequence problem and evaluated generated review comments and revised code [@p14_li2022_codereviewer]. Earlier automation work showed that review comments and revisions can be mined from review systems, but also that mined comments are heterogeneous: some ask for changes, some explain decisions, some discuss design, and some are not directly actionable [@p52_tufano2021_automating_code_review_activities]. Later qualitative analysis of code review automation further showed that aggregate performance metrics can hide which review scenarios a technique handles well and where it fails [@p53_tufano2024_code_review_automation_strengths_weaknesses].

These foundations set the boundary for our review of LLM-based work. If code review is a socio-technical practice, then a generated comment should be evaluated not only by whether it resembles a reference or flags a defect, but also by whether it supports useful review work. This is the reason later sections distinguish correctness from usefulness, actionability, workflow impact, and knowledge-transfer preservation.

## Evaluation Metrics and Benchmarks for Generated Review Comments

A central theme in prior work is that lexical similarity is insufficient for evaluating review comments. Review comments are diverse, context-dependent, and often semantically valid even when they differ from a single human reference. CodeReviewer already noted limitations of BLEU-style metrics for review-comment generation [@p14_li2022_codereviewer]. DeepCRCEval advances this line by proposing code-review-specific evaluation dimensions rather than relying only on generic text-similarity metrics [@p01_lu2025_deepcrceval]. Curated-review and preference-data studies further reinforce that review quality involves relevance, clarity, conciseness, civility, usefulness, and other dimensions that cannot be captured by exact reference matching alone [@p18_bensghaier2025_curated_reviews; @p30_weyssow2025_codeultrafeedback].

This line of work is necessary but not sufficient for our problem. A richer quality rubric can say whether a generated comment is relevant, grounded, useful, or clear. It does not by itself specify what should happen when dimensions disagree. For example, a comment may be technically plausible but weakly grounded, grounded but low-value, useful but too vague to show as-is, or relevant but costly to verify. A decision-oriented review assistant needs a policy for such cases, not only a quality score.

Recent benchmarks have moved from isolated hunk-level generation toward richer settings. SWE-PRBench and SWRBench evaluate review behavior at the pull-request or software-engineering-review level, connecting review comments to broader change context and issue coverage [@p04_kumar2026_swe_prbench; @p05_zeng2025_swrbench]. ContextCRBench emphasizes enriched semantic and code context for more realistic review evaluation [@p06_hu2025_contextcrbench]. CodeReviewQA adds diagnostic comprehension probes for review-related understanding [@p17_lin2025_codereviewqa]. These benchmarks improve realism and help evaluate issue coverage, comprehension, and context use.

However, benchmark realism does not fully solve the operational evaluation problem. A benchmark score may indicate model ranking, issue coverage, contextual adequacy, or similarity to expected feedback, but it does not directly answer whether a concrete generated comment should be shown to a developer. Our framework therefore treats benchmark results and rubric scores as evidence used by a downstream decision process. The evaluation target is not only model performance; it is the reliability and value of comment-level decisions under uncertain context and imperfect evaluators.

## Context, Grounding, and Reviewability

A second line of work treats poor generated comments as grounding failures. HalluJudge explicitly frames hallucinated code review comments as claims that are unsupported by the available diff or review context [@p02_tantithamthavorn2026_hallujudge]. Production-oriented work similarly shows that missing language, framework, version, or surrounding-code context can lead to comments that are incorrect, irrelevant, or difficult to act on [@p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. Grounding is therefore a core dimension of comment quality, but it is also a property of the selected context boundary.

Several systems attempt to improve review generation by expanding or specializing context. Retrieval-augmented approaches provide historical examples, related code, or project-specific context [@p11_zhang2025_laura; @p20_hong2025_rag_reviewer]. Context-aware systems explore file-level or repository-level evidence and routing mechanisms [@p16_icoz2026_context_aware]. Specification-grounded review uses explicit or implicit specifications as grounding evidence [@p12_wang2025_sgcr]. Static-analysis-guided approaches use analyzer warnings as focused context for LLM-generated feedback [@p22_jaoua2025_static_analyzers; @p48_patcas2026_code_quality_issues]. These approaches show that external evidence can improve review quality when it is relevant, fresh, interpretable, and correctly integrated.

The tension is that more context is not automatically better. SWE-PRBench reports settings where additional context can degrade review performance [@p04_kumar2026_swe_prbench]. Prompting and context-aware studies show that summaries, call graphs, retrieval depth, and model capacity can change whether context helps or distracts [@p13_haider2024_prompting_finetuning; @p16_icoz2026_context_aware]. Reviewability research adds that the input change itself can be hard to review when it is large, scattered, poorly described, or lacks rationale [@p40_ram2018_reviewability]. Documentation-behavior consistency work further shows that documentation and behavior can diverge, so retrieved context may be stale or contradictory [@p49_lee2025_metamon].

These findings motivate a key distinction in our framework: context availability is not the same as context quality, and context quality is not the same as context usability. We therefore treat context as an evaluation object. Relevant dimensions include relevance, completeness, specificity, consistency, freshness, groundability, provenance, behavioral evidence, attention load, and cost. A generated comment can be correct under one context boundary, unsupported under another, and impossible to judge under a third. This perspective connects context-aware review to mitigation decisions: a weakly grounded comment may need rewriting or escalation rather than a binary correct/incorrect label.

## Data Quality, Reference Quality, and Provenance

A third line of work shows that training and evaluation data are not neutral ground truth. Human review comments mined from repositories can be vague, low-value, unclear, uncivil, redundant, irrelevant, or difficult to interpret outside the original review discussion. Too Noisy To Learn and curated-review work show that cleaning or reformulating review data can improve downstream modeling, but also risks changing intent or removing useful signals [@p08_liu2025_too_noisy; @p18_bensghaier2025_curated_reviews]. Earlier automation work similarly had to distinguish comments that request code changes from comments with other review purposes [@p52_tufano2021_automating_code_review_activities].

Reference quality also depends on provenance. Reviewer-experience-aware work suggests that reviewer history, ownership, and expertise can affect the informativeness and applicability of review comments [@p23_lin2026_reviewer_experience]. Preference datasets such as CodeUltraFeedback can scale quality feedback, but they also depend on the selected judge, preference dimensions, and labeling process [@p30_weyssow2025_codeultrafeedback]. Studies of shared ChatGPT conversations in GitHub issues and pull requests add another provenance concern: AI-generated explanations can become part of collaboration artifacts and influence review discussions without necessarily being verified [@p42_wasserbaech2024_chatgpt_github].

These studies motivate a dataset-validity layer in our framework. If a reference comment is unclear, if it does not describe a review-relevant issue, if it is linked to the wrong code, if the later revision ignored it for reasons unrelated to quality, or if the reference itself was produced or shaped by an LLM, then the evaluation instance may not support a reliable judgment. In such cases, treating the model output as simply correct or incorrect confuses model failure with data or benchmark failure.

## Human-Centered Usefulness and Workflow Signals

User studies and industrial deployments show that acceptance is not the same as usefulness. RevMate reports live user-study signals such as perceived value, acceptance, reviewer time overhead, and downstream revisions [@p07_olewicki2024_revmate]. RovoDev and BitsAI-CR add production-oriented signals such as actionability gates, code resolution, pull-request cycle time, human-comment reduction, outdated comments, survey feedback, and data flywheels [@p03_tantithamthavorn2026_rovodev; @p10_sun2025_bitsai_cr]. These signals are valuable because they connect generated comments to developer behavior and workflow effects.

Recent work on human-AI review interaction further complicates the evaluation target. Human-AI synergy studies compare human and AI-agent review in terms of feedback diversity, interaction patterns, suggestion adoption, and code-quality impact [@p26_zhong2026_human_ai_synergy]. Empirical analysis of code review agents in pull requests suggests that automation can produce low-signal interactions and may be associated with weaker merge or abandonment outcomes in some settings [@p27_chowdhury2026_industry_claims]. Human-centered design work argues that AI should support review rather than fully automate it, because review also provides knowledge transfer, team awareness, shared ownership, and accountability [@p28_heander2025_support_not_automation].

These findings support separating technical correctness from usefulness, actionability, value-to-time, trust, and workflow impact. A rejected generated comment may still provide useful insight. An accepted comment may be low-value or even increase future maintenance cost. A technically correct comment may be too obvious to justify interrupting the reviewer. Therefore, a review assistant should not be evaluated only by whether its comments are plausible or accepted; it should also be evaluated by whether it preserves review value and avoids harmful noise.

## Problematic Comment Types

The literature identifies many problematic comment types, but they are often studied in separate settings. Evaluation and hallucination papers discuss unsupported claims, hallucinated issues, context-misaligned comments, incorrect assumptions, irrelevant comments, vague comments, wrong-location comments, and non-actionable feedback [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification]. Data-quality papers add noisy references, unclear comments, low-quality comments, and comments whose original intent is difficult to recover [@p08_liu2025_too_noisy; @p18_bensghaier2025_curated_reviews]. Explanation-focused work shows that generated feedback may identify a plausible concern but fail to provide a useful or grounded rationale [@p25_yu2024_finetuning_acr; @p41_widyasari2025_explaining_explanations].

Specialized settings introduce additional failure types. Secure review requires attention to vulnerability category, location, exploitability, severity, false alarms, missed vulnerabilities, and repair correctness [@p21_peng2025_icodereviewer; @p46_zhou2025_vulnerability_repair]. Static-analysis-guided review introduces failures where a warning is misread, overstated, or converted into an invalid repair suggestion [@p22_jaoua2025_static_analyzers; @p48_patcas2026_code_quality_issues]. Non-functional review requires evidence about workload, benchmarks, and maintainability trade-offs; otherwise, a performance comment may be unsupported or merely a premature optimization nitpick [@p50_peng2025_coffe].

Our synthesis separates problematic comment types from evaluation dimensions. For example, actionability is an evaluation dimension, while a non-actionable comment is a failure type. Grounding is a dimension, while an unsupported claim is a failure type. The taxonomy also separates four layers that are often conflated: generated-review failures, input or context failures, workflow failures, and evaluator failures. This separation matters because different failures require different mitigation strategies. A vague but relevant comment may need rewriting; a clearly hallucinated comment should be suppressed; a weakly grounded but potentially important security concern may need escalation; and a biased judge decision should trigger evaluator calibration rather than a model-quality conclusion.

## Mitigation Strategies and Their Trade-offs

Prior work has explored several mitigation strategies, but they operate at different points in the review pipeline. Pre-generation strategies include data cleaning, reviewability checks, prompt routing, retrieval, specification grounding, and static-analysis context construction [@p08_liu2025_too_noisy; @p11_zhang2025_laura; @p12_wang2025_sgcr; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers]. Generation-time strategies include fine-tuning, reward-guided optimization, multi-agent generation, and structured output generation [@p09_ren2025_hydra_reviewer; @p24_bensghaier2025_reward_models; @p25_yu2024_finetuning_acr]. Post-generation strategies include hallucination detection, actionability and relevance filters, LLM critics, rewriting, suppression, and human escalation [@p02_tantithamthavorn2026_hallujudge; @p03_tantithamthavorn2026_rovodev; @p10_sun2025_bitsai_cr; @p35_mcaleese2024_llm_critics].

This distinction matters because each mitigation changes a different trade-off. A pre-generation context gate can avoid unreliable review on low-quality inputs, but it can also skip useful automation opportunities. Retrieval can improve grounding while increasing token cost, latency, stale-context risk, and attention load. Specification grounding can improve trust, but only if specifications remain current and consistent with code. Static-analysis context can focus the model, but tool warnings can be false, irrelevant, or misinterpreted. Post-generation hallucination or relevance gates can reduce bad comments while suppressing rare but useful feedback. Human escalation can improve safety, but it adds delay and reviewer overhead. Critic-assisted review can increase comprehensiveness, but may also introduce hallucinated bug reports or low-value nitpicks [@p35_mcaleese2024_llm_critics].

The under-measured quantity across these strategies is useful-feedback preservation. Many papers measure better scores, lower hallucination, higher acceptance, or fewer low-quality comments. Fewer papers measure the useful comments that are wrongly removed, the human-flagged issues no longer covered after filtering, or the value lost when automation reduces human discussion. A trade-off-aware evaluation should therefore report both error-reduction metrics and preservation metrics: harmful comments removed, useful comments retained, useful comments wrongly suppressed, review coverage retained after filtering, valuable-but-not-accepted comments preserved, human escalation rate, and the cost of verification. This is the trade-off our framework makes explicit: mitigation should be evaluated by what it removes, what it preserves, what it accidentally loses, and what it costs.

## LLM-as-a-Judge and Evaluator Validity

LLM-as-a-Judge methods are increasingly used to evaluate software-engineering artifacts. They can reduce manual evaluation cost, support scalable comparison, and produce structured judgments. However, recent work argues that LLM judges should be treated as measurement instruments rather than ground truth [@m04_zheng2023_llm_judge; @p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p33_he2025_llmjudge_se; @p36_li2024_llms_as_judges].

Several risks are especially relevant to code review evaluation. Judge behavior can depend on task type, prompt wording, response order, source model, answer verbosity, preprocessing, output format, and other biases [@p29_wang2025_human_evaluators; @p30_weyssow2025_codeultrafeedback; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se; @p36_li2024_llms_as_judges]. If an LLM judge is used to filter comments, score usefulness, compare systems, or decide whether a comment should be escalated, these sensitivities become part of the evaluation result. Without calibration, repeated-run checks, answer-rate reporting, order-swap tests, prompt-perturbation analysis, and human agreement, evaluation may replace weak lexical metrics with a different weak proxy.

This paper therefore separates evaluator validity from comment quality. A generated comment can be useful while an evaluator fails to recognize it, and a harmful comment can be rated as plausible by an uncalibrated judge. Judge outputs are evidence that must be validated, not final truth. In the proposed framework, evaluator failures are part of the evaluation problem rather than external noise to be ignored.

## Specialized Evaluation Sublayers

Some review tasks require specialized evaluation layers. Security review has asymmetric costs: false alarms can erode trust, while missed vulnerabilities may have severe consequences. It therefore requires structured validation of vulnerability type, location, exploitability, severity, and repair correctness [@p21_peng2025_icodereviewer; @p46_zhou2025_vulnerability_repair]. Static-analysis and code-quality review require validation that a warning has been interpreted correctly and that the proposed change resolves the issue without changing intended behavior [@p22_jaoua2025_static_analyzers; @p48_patcas2026_code_quality_issues]. Efficiency and performance comments require workload and benchmark evidence, because generic performance advice may be unsupported or harmful to maintainability [@p50_peng2025_coffe].

These specialized layers should not be collapsed into a single generic quality score. They extend the general framework with task-specific evidence requirements. A general review-comment framework should therefore allow optional sublayers for security, static-analysis-guided quality issues, repair validation, documentation consistency, and non-functional claims.

## Summary of the Gap

Prior work has introduced better benchmarks, richer evaluation rubrics, hallucination detection, context-aware generation, industrial filters, static-analysis hybrids, reward and critic mechanisms, and LLM-as-a-Judge evaluation. It has also advanced data curation, reference-quality analysis, fine-grained comment classification, explanation studies, and human-centered usefulness evaluation. The gap is not the absence of these dimensions. The gap is that they are not yet fully integrated around the operational decision a review assistant must make about each generated comment.

The closest prior work covers parts of this space: quality rubrics improve comment-level scoring, hallucination work improves grounding checks, PR-level benchmarks improve realism, production systems add workflow signals, context-enriched systems add evidence, data-quality work improves reference and dataset reliability, classification and explanation studies support taxonomy construction, and LLM-as-a-Judge studies improve evaluator scalability. What remains missing is a unified account of how these signals should be combined when deciding whether a generated review comment should be shown, suppressed, rewritten, or escalated.

Our work addresses this integration gap by proposing an operational taxonomy of problematic generated review comments and a trade-off-aware evaluation framework. The framework connects generated-comment quality, context quality, reference and provenance quality, usefulness, actionability, workflow impact, mitigation decision, useful-feedback preservation, specialized sublayers, cost, and evaluator validity. The intended outcome is not a single universal score or another benchmark leaderboard, but a structured way to reason about which errors are reduced, which useful signals are preserved, what costs are introduced, and which comments should be shown, suppressed, rewritten, or escalated.
