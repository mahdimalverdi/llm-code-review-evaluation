# Related Work

This section positions the proposed taxonomy and framework within prior work on code review automation, LLM-based review assistance, evaluation benchmarks, context grounding, data quality, human-centered usefulness, mitigation strategies, and evaluator validity. Rather than reviewing papers one by one, we organize the literature by the evaluation dimension each line of work emphasizes. This organization supports the main claim of the paper: prior work has made substantial progress on individual dimensions, but these dimensions are not yet fully integrated around the downstream decision of whether a generated review comment should be shown, suppressed, rewritten, or escalated.

## Evaluation of Generated Code Review Comments

Early work on automated code review helped establish review-comment generation and review-driven code transformation as model tasks. CodeReviewer, for example, framed code review as a sequence-to-sequence problem and studied model-generated review comments and code changes [@p14_li2022_codereviewer]. Pre-LLM automation work also showed that review comments and revised code can be mined from review systems, but that exact-match and lexical metrics are incomplete because alternative outputs may still be semantically useful [@p52_tufano2021_automating_code_review_activities]. Later qualitative analysis of code review automation further showed that aggregate metrics can hide which kinds of review scenarios a technique can or cannot handle [@p53_tufano2024_code_review_automation_strengths_weaknesses].

Recent work has moved toward evaluation settings that are closer to real review workflows. DeepCRCEval argues for domain-specific evaluation of generated code review comments rather than relying only on generic text-similarity metrics [@p01_lu2025_deepcrceval]. SWE-PRBench and SWRBench move evaluation toward pull-request-level or software-engineering-review settings, where review comments are connected to broader change context and human-flagged issues [@p04_kumar2026_swe_prbench; @p05_zeng2025_swrbench]. ContextCRBench emphasizes contextual information and benchmark realism for code review evaluation [@p06_hu2025_contextcrbench]. These benchmarks are important because they broaden the evaluation target beyond isolated hunks. However, benchmark realism alone does not determine which generated comments should be shown to developers. A score can indicate issue coverage, similarity, or quality, but a review assistant still needs a decision policy for comments that are uncertain, partially grounded, low-value, or useful only after rewriting.

This paper builds on this benchmark and metric work, but shifts the unit of analysis from model performance alone to decision-oriented evaluation. The question is not only whether a system generates a good comment, but whether the evaluation signals are sufficient to decide how the comment should be handled in a review workflow.

## Context, Grounding, and Hallucination

A major line of work treats poor review comments as failures of grounding. HalluJudge explicitly studies hallucinated code review comments and evaluates whether generated comments are supported by the available context [@p02_tantithamthavorn2026_hallujudge]. Industrial and production-oriented systems also show that missing or incomplete context can lead to comments that are incorrect, irrelevant, or difficult for developers to act on [@p03_tantithamthavorn2026_rovodev; @p07_olewicki2024_revmate]. These studies are directly relevant to our taxonomy because unsupported or hallucinated comments are among the most visible classes of problematic generated review comments.

Other work expands the kinds of context that review assistants may use. Retrieval-augmented and context-aware systems incorporate surrounding code, related files, historical artifacts, or project-specific information [@p11_zhang2025_laura; @p16_icoz2026_context_aware; @p20_hong2025_rag_reviewer]. Specification-grounded review adds external requirements or expected behavior as grounding evidence [@p12_wang2025_sgcr]. Static-analysis-guided review uses analyzer findings as focused evidence for generated feedback [@p22_jaoua2025_static_analyzers]. Work on documentation-behavior consistency also shows that retrieved or attached context can be stale, contradictory, or behaviorally misaligned [@p49_lee2025_metamon].

These studies show that context is not simply a matter of adding more input. Additional context can improve grounding, but it can also add noise, increase cost, and make evaluator attention more difficult to manage. Reviewability studies further show that some changes are hard to review because the input change itself is large, scattered, poorly described, or missing rationale [@p40_ram2018_reviewability]. Therefore, this paper treats context quality as an evaluation dimension. A generated comment may be correct under one context boundary, unsupported under another, and impossible to judge under a third. This perspective connects grounding work to mitigation decisions: a weakly grounded comment may need suppression, rewriting, or escalation rather than a binary correct/incorrect label.

## Data Quality, Reference Quality, and Provenance

Another line of work shows that review data and reference comments are not clean ground truth. Review comments mined from repositories can be vague, unclear, non-actionable, irrelevant, noisy, or difficult to interpret outside the original review discussion. Too Noisy To Learn and curated-review work show that cleaning, filtering, or reformulating review data can improve downstream modeling, but also raises the risk of removing useful signals or changing the original review intent [@p08_liu2025_too_noisy; @p18_bensghaier2025_curated_reviews]. Earlier automation work similarly had to filter review comments because not every comment linked to code asks for a code change [@p52_tufano2021_automating_code_review_activities].

Reference quality also depends on provenance. Reviewer-experience-aware work suggests that the source and expertise behind a review comment can affect its interpretation and value [@p23_lin2026_reviewer_experience]. Preference-data and feedback datasets, including CodeUltraFeedback, show that large-scale annotation or preference construction can support evaluation, but they also introduce dependence on the labeling process and the evaluator used to produce labels [@p30_weyssow2025_codeultrafeedback]. Studies of shared ChatGPT conversations in GitHub issues and pull requests add another provenance concern: AI-generated explanations can become part of collaboration artifacts and influence review discussions without necessarily being verified [@p42_wasserbaech2024_chatgpt_github].

These findings motivate the dataset-validity dimension in our framework. If a reference comment is unclear, if it does not ask for a review-relevant action, if the later revision ignored it, or if it is linked to the wrong code, then the evaluation instance itself may not support a reliable judgment. In such cases, treating the model output as simply correct or incorrect can confuse model failure with dataset or benchmark failure.

## Human-Centered Usefulness and Workflow Signals

Human code review studies show that review value goes beyond defect detection. Modern code review supports maintainability, knowledge transfer, team awareness, shared ownership, alternative solution discovery, and developer learning [@p37_sadowski2018_google_mcr; @p38_bacchelli2013_expectations_mcr; @p39_bosu2015_useful_reviews; @p51_davila2021_mcr_slr_taxonomy]. This matters for LLM-based review assistance because a generated comment can be technically correct but still low-value, distracting, or harmful to the human review process. Conversely, a comment that does not identify a definite bug may still be useful if it improves understanding or triggers a valuable design discussion.

User studies and industrial evaluations reinforce the distinction between acceptance and usefulness. RevMate, production deployments, and human-AI review-support studies show that developer interaction with generated feedback depends on perceived value, effort, trust, workflow fit, and reviewer burden [@p07_olewicki2024_revmate; @p10_sun2025_bitsai_cr; @p26_zhong2026_human_ai_synergy; @p27_chowdhury2026_industry_claims; @p28_heander2025_support_not_automation]. Acceptance, rejection, or editing of a generated comment is therefore only one signal. A rejected comment may still provide useful insight, while an accepted suggestion may be low-value or even increase future maintenance cost.

This paper uses these human-centered findings to separate correctness from usefulness and workflow impact. The goal is not to replace human-centered evaluation with automatic scoring, but to connect human value signals to operational decisions. In a review assistant, the relevant question is whether a comment should be shown as-is, rewritten to reduce uncertainty, suppressed to reduce noise, or escalated because it may be important but difficult to judge automatically.

## Problematic Comment Types and Mitigation Strategies

The literature identifies many problematic comment types, but they are often studied in separate contexts. Evaluation and hallucination papers discuss unsupported claims, hallucinated issues, irrelevant comments, wrong assumptions, non-actionable feedback, and wrong-location comments [@p01_lu2025_deepcrceval; @p02_tantithamthavorn2026_hallujudge; @p19_nguyen2025_fine_grained_classification]. Data-quality papers add unclear comments, noisy references, and low-signal comments [@p08_liu2025_too_noisy; @p18_bensghaier2025_curated_reviews]. Explanation-focused and code-quality work further suggests that a comment may fail by providing an unhelpful explanation, misrepresenting severity, or giving advice that is too generic for the code under review [@p41_widyasari2025_explaining_explanations; @p48_patcas2026_code_quality_issues].

Mitigation strategies are similarly diverse. Prior work has explored post-generation hallucination detection, relevance or actionability filters, retrieval-augmented generation, specification grounding, static-analysis hybrids, prompt routing, multi-agent generation, reward models, fine-tuning, LLM critics, and human-in-the-loop workflows [@p02_tantithamthavorn2026_hallujudge; @p09_ren2025_hydra_reviewer; @p10_sun2025_bitsai_cr; @p11_zhang2025_laura; @p12_wang2025_sgcr; @p21_peng2025_icodereviewer; @p22_jaoua2025_static_analyzers; @p24_bensghaier2025_reward_models; @p25_yu2024_finetuning_acr; @p35_mcaleese2024_llm_critics]. These strategies can reduce some failure modes, but they are not free improvements. A stricter filter can suppress hallucinated comments but also remove rare useful comments. More context can improve grounding but increase latency, cost, and attention load. Static-analysis context can provide focused evidence but may be misinterpreted by the model. Human escalation can improve reliability but adds delay and reviewer overhead.

Security, static-analysis, and non-functional-review settings make the trade-offs even more visible. Secure code review requires attention to vulnerability type, location, exploitability, severity, repair correctness, false-positive cost, and missed-vulnerability risk [@p21_peng2025_icodereviewer; @p46_zhou2025_vulnerability_repair]. Static-analysis-guided review requires warning interpretation and behavior preservation [@p22_jaoua2025_static_analyzers]. Non-functional quality comments require evidence about performance, efficiency, workload assumptions, and maintainability trade-offs [@p48_patcas2026_code_quality_issues; @p50_peng2025_coffe]. These specialized sublayers support the need for a framework that can include task-specific criteria without collapsing all evaluation into a single generic quality score.

## LLM-as-a-Judge and Evaluator Validity

LLM-as-a-Judge is increasingly used to evaluate software-engineering artifacts, including generated code, explanations, and review feedback. It can reduce manual evaluation cost and provide scalable judgments, but recent work warns that LLM judges should be treated as measurement instruments rather than ground truth [@m04_zheng2023_llm_judge; @p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p33_he2025_llmjudge_se; @p36_li2024_llms_as_judges].

Several risks are especially relevant to code review evaluation. Judge behavior can depend on prompt wording, response order, answer verbosity, source model, preprocessing, and task framing [@p29_wang2025_human_evaluators; @p31_jiang2025_codejudgebench; @p32_zhao2026_bias_loop; @p33_he2025_llmjudge_se]. If an LLM judge is used to filter comments, score usefulness, compare systems, or select comments for human review, these sensitivities can change the measured quality of a system. Without calibration, repeated-run checks, human agreement, and sensitivity analysis, an evaluation may replace one weak proxy metric with another.

This paper therefore treats evaluator validity as a separate framework dimension. The reliability of the evaluator is not the same as the quality of the generated comment. A generated comment can be useful while the evaluator fails to recognize it, or harmful while the evaluator rates it as plausible. This separation is necessary for any framework that uses LLM-based evaluation as part of a mitigation pipeline.

## Summary of the Gap

Prior work has made strong progress on evaluating LLM-based code review systems and related software-engineering artifacts. It has introduced better benchmarks, richer rubrics, hallucination detection, context-aware generation, data curation, human-usefulness studies, mitigation strategies, and LLM-as-a-Judge evaluation. The gap is not the absence of these dimensions. The gap is that they are not yet fully connected to the operational decision that a review assistant must make about each generated comment.

Our work addresses this integration gap by proposing an operational taxonomy of problematic generated review comments and a trade-off-aware evaluation framework. The framework connects comment quality, context quality, dataset validity, usefulness, actionability, workflow impact, cost, mitigation decisions, specialized sublayers, and evaluator validity. The intended outcome is not a single universal score, but a structured way to reason about which errors are reduced, which useful signals are preserved, what costs are introduced, and which comments should be shown, suppressed, rewritten, or escalated.
